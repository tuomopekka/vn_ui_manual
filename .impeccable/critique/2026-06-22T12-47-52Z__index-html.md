## Anti-Patterns Verdict

**LLM assessment**: Still not AI-slop in the visual sense — no gradients, no glassmorphism, no marketing throat-clearing. The "Visuaalinen hahmotelma" wireframe-labeling convention is a genuine strength: it tells a zero-context vendor unambiguously what's illustrative vs. spec. The tells that remain are seams, not surface decoration: a cookie-banner demo whose markup contradicts its own adjacent spec table, an orphaned `::after` CSS rule that styles nothing visible, and hundreds of inline `style=""` attributes instead of utility classes — the signature of fast, unrefactored production rather than a hand-tuned system.

**Deterministic scan**: 1 finding (exit 2) — **em-dash overuse**, detector-reported count **49**, but a direct Python count of the real `—` Unicode character returns **29**. The ~20-occurrence gap is `var(--space-xs)`-style double-hyphen tokens inside inline `style="..."` attributes being swept into the detector's body-text extraction — a tool-side false-positive contributor, not real content (confirmed mechanically, not just asserted). `numbered-section-markers` did not fire, confirming the persisted ignore-rule is correctly scoped. The 29 real em-dashes are the same set already reviewed in the prior `/impeccable clarify` pass and judged legitimate Finnish explanatory-clause usage, not re-flagged here.

**Verified fixes from the previous pass** (independently re-checked by both assessments, not assumed): anchor integrity (25 hrefs / 39 ids, zero broken, zero duplicates), dead sidebar-toggle/backdrop/drawer code (zero remaining references), alert roles (`status` ×3, `alert` ×1, correctly assigned), search form (`role="search"` present), ministry filter (input + working JS + empty state), `.form-error`/`.required-star` using the new contrast-safe token, `.nav-lang-btn` 44×44px touch target, `prefers-reduced-motion` override on `scroll-behavior`, and the trimmed 4-combo font payload. All hold.

**Visual overlays**: Unavailable — no browser automation tool in this environment.

## Overall Impression

The fix pass genuinely worked — every item from the last critique (P0 mobile-nav non-issue resolved by author confirmation, both P1s, all P2/P3s) is independently verified fixed, with no regressions. But a fresh, independent design pass — deliberately not trusting the prior fix list — surfaced new issues the previous pass didn't catch: a cookie-banner demo that contradicts its own accessibility spec one table-row away, a systemic gap where none of the document's 27 `.table-wrap` instances carry the `role="region"`/`tabindex="0"` that design.md §10.2 itself mandates, and a sidebar group that's still double the page's own stated chunking guidance. The pattern from the original audit — "demo violates the rule stated next to it" — recurs here in a new spot, suggesting it's worth a dedicated sweep rather than one-off patches each time a critique catches the next instance.

## Priority Issues

**[P1] Cookie banner demo markup contradicts its own accessibility spec.**
Why it matters: The requirements table (line 1680) states the banner is implemented as `role="dialog"`, but the actual demo div (line 1657, `<div class="cookie-banner-demo">`) has no `role` attribute at all — verified directly. Cookie banners are universal and legally mandated; a vendor who copies the visible markup ships a non-conformant banner while believing they followed spec, because the prose immediately next to the code says otherwise. This is the exact failure mode PRODUCT.md principle #1 calls unacceptable.
Fix: Add the role to the actual demo element — and first settle whether `role="dialog"` (implies a focus-trapping modal) or `role="region"` with `aria-label` (correct for a non-modal, content-pushing banner per design.md §2.8) is actually right, since the doc currently doesn't specify which.
Suggested command: /impeccable audit

**[P1] None of the document's 27 `.table-wrap` instances carry the `role="region"`/`tabindex="0"` design.md §10.2 itself requires for scrollable tables.**
Why it matters: Verified directly (`grep -c 'class="table-wrap"'` → 27; zero combine it with `role="region"`). This is the exact "reference tool fails its own documented accessibility rule" pattern already fixed once for alerts and search — now found in every single table on the page. A vendor copying any table loses keyboard/screen-reader access to horizontally-scrolling content.
Fix: Add `role="region" tabindex="0" aria-label="..."` to the `.table-wrap` wrapper, ideally via a small JS pass or by auditing the pattern once and templating it, rather than 27 manual edits.
Suggested command: /impeccable harden

**[P2] Sidebar "Visuaaliset tokenit" group still has 8 items — double the page's own 5-7-item nav guidance, and the densest, most-referenced part of the document.**
Why it matters: Independently confirmed by both assessments. This is colors, ministry palette, type, spacing, radius, states, and logos — exactly where a time-pressed vendor needs the fastest scan, and the one nav-group the prior fix pass didn't touch (it split "Komponentit" but left this one as-is).
Fix: Split into two groups (e.g. "Värit ja ministeriöt" / "Typografia ja muut tokenit") or add visual sub-indentation so "3.1", "3.1.4", etc. read as children of "3." rather than flat siblings.
Suggested command: /impeccable layout

**[P2] Select-element focus/default arrow icons are hardcoded to `#5D626A`/`#365ABD` hex in an inline SVG data-URI, not the ministry color token.**
Why it matters: Verified at design-system.css:748,762. This breaks the "one shared system, ministry-tinted" premise design.md promises — copying this CSS verbatim into a ministry whose `--ministry-color` differs from the default VN blue leaves a focus-state arrow that silently doesn't track the theme.
Fix: Generate the SVG data-URI dynamically or accept the small inconsistency explicitly as a documented exception (most ministry select-arrows are decorative enough that an exact-match isn't critical, but it should be a stated decision, not an oversight).
Suggested command: /impeccable harden

**[P3] Orphaned CSS: `.doc-header h1::after` (line 282-284) styles a pseudo-element that never renders, since `.doc-header h1` never gets the width/height/display treatment that makes the identical pattern work for `.section-title::after`.**
Why it matters: Low visual impact, but it's the kind of unverified-against-the-rendered-page CSS that, multiplied, produces a "subtly off" feeling on close inspection.
Fix: Either complete the rule to match `.section-title::after`'s working pattern, or delete it.
Suggested command: /impeccable harden

**[P3] Document ends abruptly after print styles — no closing beat, no back-to-top, no pointer to design.md's own changelog/ownership section (§20).**
Why it matters: Per the peak-end rule, the last impression colors recall of the whole tool; ending on print-page-number trivia undersells a document that opened with real authority.
Fix: Add a minimal closing element — a "Takaisin alkuun" link plus a one-line pointer to the changelog/ownership info design.md §20 already contains but the HTML never surfaces.
Suggested command: /impeccable harden
