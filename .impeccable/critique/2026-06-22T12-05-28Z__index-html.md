## Anti-Patterns Verdict

**LLM assessment**: Not "AI slop" in the cheap-tell sense — no gradients, no glassmorphism, no marketing puffery, the visual register matches the official/calm/restrained brief well. The real tell is structural: a 1844-line file built almost entirely by repetition-with-substitution (12 near-identical ministry color blocks, dozens of `subsection--wireframe` wrappers, ~25 near-identical table wrappers used inconsistently for content that sometimes doesn't deserve a table). A styled-but-never-used `.sidebar-filter-input` class is the inverse tell — a feature scaffolded then abandoned mid-build.

**Deterministic scan**: `detect.mjs --json index.html` → exit code 2, 2 findings:
- **Em-dash overuse** (warning): 72 em-dashes in body copy — an AI cadence tell the design-review pass didn't separately flag. No claim made about which are legitimate Finnish formal-register usage vs. cadence artifact; flagged as-is.
- **Numbered section markers** (advisory): sequence 05, 10, 11, 12 — **ruled out as a false positive** by both assessments independently. These are design.md's own canonical section numbers (`<span class="section-number">N.</span>` inside each `<h2>`), required structural TOC numbering for a reference doc, not a decorative AI eyebrow pattern.

Both assessments independently verified the anchor-integrity and table-markup issues flagged in the *previous* critique run (2026-06-18) are now fixed: all 20 sidebar anchors resolve correctly with no duplicates, and `scope="row"` appears only on `<th>`, never on `<td>`. Real, verified progress since last time.

**Visual overlays**: Unavailable — no browser automation tool in this environment. CLI scan + structural review only, no live screenshot evidence.

## Overall Impression

This page has genuinely improved since the last critique: the broken anchor and the misplaced `scope="row"` are both fixed, and the sidebar now does real scroll-spy instead of a hardcoded active state. The visual register is right — calm, restrained, faithful to design.md. The sidebar/TOC being hidden on mobile is a confirmed intentional design decision, not a bug, and is not scored as an issue. The remaining gaps are narrower: two P1s around alert accessibility and ministry-color lookup, plus an uneven sidebar grouping that the page's own §6.2 rule would flag.

## What's Working

1. **The focus-color rationale (index.html:110)** — rather than silently picking a hex, the copy explains the brand orange fails 3:1 contrast and derives a separate, darker, compliant focus token. Exactly the kind of transparent reasoning a zero-context vendor needs to trust the system.
2. **Real `IntersectionObserver` scroll-spy** (lines 1760-1792), replacing the hardcoded `is-active` flagged last time — genuinely fixed, not papered over.
3. **Print stylesheet depth** (design-system.css:880-914) — per-element page-break control, `-webkit-print-color-adjust: exact` on swatches/table headers, explicit margin handling. Most teams skip this; this one didn't.

## Priority Issues

> Note: the original pass flagged the hidden mobile sidebar/toggle as a P0 bug. Confirmed by the author as an intentional design decision (navigation deliberately disabled on mobile) — removed from issues below, not a defect.

**[P1] Alert/huomionauha demos are missing the `role="alert"`/`role="status"` design.md §13.1 explicitly mandates.**
Why it matters: The four alert boxes (lines 1557-1572: info/warning/error/success) have no `role` attribute at all, despite design.md stating plainly that urgent messages need `role="alert"` and others `role="status"`. This is a copy-paste reference; as shown, copying it verbatim ships a non-compliant component on a digipalvelulaki-mandated point — exactly the failure mode PRODUCT.md's principle #1 calls unacceptable.
Fix: Add `role="status"` to the three non-urgent variants and `role="alert"` to the error variant; note in the adjacent spec table why they differ.
Suggested command: /impeccable audit

**[P1] No way to jump to a single ministry's colors in the 12-ministry palette.**
Why it matters: §3.1.4 renders all twelve ministries' full 5-step color ramps (60 swatches) unconditionally and sequentially, with no filter or jump-list. "Find my ministry's accent color" is the single most likely task for an external vendor under deadline pressure, and it's currently solved only by scrolling past 10+ unrelated palettes. A `.sidebar-filter-input` class exists in the CSS (design-system.css:187-198) fully styled but unused in markup — this was apparently considered and dropped mid-build.
Fix: Wire up the existing filter-input styling to a simple ministry jump-list/filter above the color grid, or at minimum add a compact per-ministry index at the top of the section.
Suggested command: /impeccable layout

**[P2] Sidebar nav-group sizes are wildly uneven (2 / 7 / 1 / 11 items), violating the page's own §6.2 navigation guidance.**
Why it matters: The "Komponentit" group bundles 11 sidebar entries into one undifferentiated group, while design.md's own §6.2 recommends capping nav at 5-7 items "too many items weakens readability and usability." The reference document fails to apply its own rule to itself — a credibility problem for a source-of-truth doc, and a chunking failure (working-memory load) for anyone scanning the sidebar.
Fix: Split "Komponentit" into 2-3 subgroups (e.g. Navigation & Chrome / Content Components / Feedback & Forms) of 3-5 items each.
Suggested command: /impeccable layout

**[P3] Section `id` values drift from displayed section numbers past §12 (e.g. `id="s13"` renders as "14.", `id="s13-media"` renders as "13.").**
Why it matters: Harmless today (every link was independently verified to resolve correctly, no duplicates), but a maintenance trap — the next person extending this file via id-based search/replace will edit the wrong section.
Fix: Rename ids to track displayed numbers (`id="s13"` → `id="s14-alerts"`, etc.).
Suggested command: /impeccable harden

**[Detector, advisory] 72 em-dashes in body copy flagged as an AI cadence tell.**
Why it matters: Not confirmed as wrong for this register (formal Finnish prose uses em-dashes legitimately), but worth a copy pass given the volume.
Suggested command: /impeccable clarify
