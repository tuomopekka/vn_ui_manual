## What's Working

1. **The ministry-filter live-region pattern is engineered correctly, not just superficially fixed** — always-present element, `textContent`-only updates, CSS `:empty` for visual hiding. This avoids the announce-then-immediately-remove race condition that plagues naive `hidden`-toggling implementations, and it's rare to see done right even in production code.
2. **The §3.1.4 ministry-color caveat models honest documentation practice** — answering "are these literally what I should hardcode?" before a skeptical vendor has to ask, directly serving PRODUCT.md's "design for the zero-context vendor" principle.
3. **Scroll-spy degrades gracefully**: feature-detects `IntersectionObserver`, bails cleanly if absent, and tracks a `visible` set rather than a single observed target — avoiding the flicker bug where two adjacent thin sections both register "visible."

## Priority Issues

**[P2] No copy-to-clipboard on hex/token values.**
Why it matters: PRODUCT.md frames the job-to-be-done as "look up the correct token... and implement it correctly the first time" — a copy-paste action. The highest-frequency instance of that action (12 ministries × 5 color steps = 60 hex values, plus every CSS variable name) has no accelerator anywhere in the document.
Fix: Add a click-to-copy button on `.swatch-hex` and `<code>` token elements, with a brief status announcement reusing the same live-region pattern just built for the ministry filter.
Suggested command: /impeccable delight

**[P2] Static error/disabled/destructive demo states have no "this is an example" cue.**
Why it matters: The §8 email field shows a permanent error; §7's destructive/disabled buttons are permanently in those states. A time-pressed vendor skimming quickly could momentarily read the always-red email field as the page itself being broken — the wireframe label "Visuaalinen hahmotelma" doesn't specifically flag "this is deliberately showing an error state."
Fix: Add a small muted annotation near static error/disabled examples, e.g. "Esimerkki virhetilasta," matching the explanatory-caption convention already used at §3.1.4.
Suggested command: /impeccable clarify

**[P3] No inline cross-references between sections governed by the same rule.**
Why it matters: §3.1.5 (error/warning/focus colors) doesn't link to §2's contrast rule; §8's error guidance doesn't link to §14's alert-role guidance, even though both implement the same "color + icon + text" redundancy principle. Minor — easy workaround via sidebar/search.
Fix: Add short "(ks. osio 2.4)"-style parenthetical links, matching design.md's own internal-reference convention.
Suggested command: /impeccable clarify

**[P3] No "how to use this document" orientation for first-time vendors.**
Why it matters: The header states what the document is but not how a new reader should approach it. Jordan (zero-context vendor) lands cold with no signal about reading order beyond the sidebar's implicit structure.
Fix: One sentence in the header description pointing new readers to §1–2 first, others straight to their component.
Suggested command: /impeccable onboard

**[P3] Generic icon roster in §12 doesn't echo the document's own running example.**
Why it matters: Minor "show, don't just tell" miss — the icons shown are the default starter set rather than ones tied to the rakennuslupa/permit thread running through §6–11.
Fix: Swap 2-3 icons for ones already implied elsewhere (e.g. `description` or `picture_as_pdf` matching the §7 download-link demo).
Suggested command: /impeccable delight
