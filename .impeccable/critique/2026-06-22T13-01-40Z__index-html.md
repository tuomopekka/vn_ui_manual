## Priority Issues

**[P2] Inconsistent sidebar promotion under §3.1 — some subsections get nav entries, others don't, with no visible rule.**
Why it matters: Verified directly — `id="s3-colors-sub"` (3.1 Väripaletti) contains three real `<h3>` headings with no ids and no sidebar links (3.1.1 Päävärit, 3.1.2 Tukivärit, 3.1.3 Taustavärit), and 3.1.5 (Virhe-, varoitus- ja ilmoitusvärit) is likewise real, headed content absent from the nav — while sibling-level 3.1 and 3.1.4 both get links. A vendor scanning the sidebar for "where are the support/error colors documented" won't find them listed and may assume they don't exist, even though they're the system's only error/warning/focus hex values.
Fix: Either give 3.1.1–3.1.3 and 3.1.5 their own sidebar entries, or add a one-line pointer under "3.1 Väripaletti" noting the subsections live below it, so the omission reads as deliberate.
Suggested command: /impeccable layout

**[P2] Ministry-filter empty-state message isn't announced to screen readers.**
Why it matters: `#ministry-filter-empty` toggles via the `hidden` attribute and is only wired through `aria-describedby` on the input — which announces on focus, not reliably on a value change while the field still has focus. design.md §15.7 already specifies `aria-live="polite"` + `aria-atomic="true"` for exactly this filter-result-count pattern elsewhere in the system; the one interactive control added in a prior fix pass doesn't yet follow it.
Fix: Add `aria-live="polite"` to the empty-state element (or a dedicated visually-hidden status node) so it's announced as it appears.
Suggested command: /impeccable harden

**[P3] "Typografia ja muut tokenit" nav-group has 5 items — satisfies design.md §6.2's literal 5-7-item rule, but exceeds the stricter ≤4 cognitive-chunking ideal applied elsewhere (every other group in the sidebar is ≤4).**
Why it matters: Minor — the document's own explicit numeric rule is met. Worth a follow-up only because it's now the one visible exception to an otherwise-consistent ≤4 pattern.
Fix: Split into "Typografia" (3.2) and "Muut tokenit" (3.4, 3.5, 3.6), or accept as-is since it complies with the stated rule.
Suggested command: /impeccable layout

**[P3] Ministry color swatches (60 hex values across 12 blocks) are hand-maintained as inline `style="background:#..."` rather than CSS custom properties, with no documented sync note — unlike the select-arrow exception, which has one.**
Why it matters: A vendor under deadline pressure could reasonably infer "inline hex is the production pattern here," since the page's own demo models it — even though design.md §17 states the opposite (themes should be dynamic CSS custom properties). Same risk class as the select-arrow case, just larger surface area, and not yet given the same explicit "this is intentional, here's why" treatment.
Fix: A one-line note near §3.1.4 acknowledging the swatches are illustrative/hand-synced from design.md's table, mirroring the select-arrow comment's pattern.
Suggested command: /impeccable harden
