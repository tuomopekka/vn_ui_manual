## Anti-Patterns Verdict

**LLM assessment**: Still no AI-slop visual tells. The copy-to-clipboard feature uses the right primitives (event delegation, single shared live region, stable `aria-label`, `::before` touch-target expansion) — more sophisticated than a typical "I added a button" AI pattern. But it has a textbook re-entrancy bug: optimized for the single happy-path click, never tested against a second click landing mid-animation. That's exactly the kind of bug that slips through when verification checks "does it work" rather than "does it survive being used the way an impatient user actually uses it."

**Deterministic scan**: 1 finding (exit 2) — em-dash overuse, detector-reported 51 vs. raw character count 31. Same established `var(--token)` noise pattern, not new content.

**Bug independently verified by me, not just relayed**: traced the click handler directly. `value` and `original` are both read from `button.textContent` *at click time*. If a button is clicked a second time while still showing "Kopioitu" (within the 1200ms restore window), `value` becomes the literal string `"Kopioitu"` — that gets written to the clipboard instead of the hex code, the live region announces nonsense ("Kopioitu väriarvo Kopioitu leikepöydälle"), and the overlapping `setTimeout` calls race: the first restores the correct hex, but the second (scheduled from the corrupted click) then overwrites it back to the literal word "Kopioitu" — permanently, until page reload. Confirmed via direct code trace, not assumption.

## Priority Issues

**[P1] Copy-to-clipboard button corrupts its own label on a second click within 1200ms.**
Why it matters: `value` and `original` are both read from the button's live `textContent` at click time rather than from a stable source. A user double-clicking (a completely normal "make sure it worked" action) within the 1200ms restore window copies the literal string "Kopioitu" to the clipboard instead of the hex code, and overlapping `setTimeout` calls race — the button's visible label can end up permanently stuck on "Kopioitu" instead of the real hex value until the page is reloaded. Verified directly via code trace, reproducible by description.
Fix: Read the hex value from a stable source (e.g. a `data-hex` attribute set once at load, or strip "Kopioi väriarvo " from the unchanging `aria-label`) instead of mutable `textContent`; track and clear any pending restore timeout before starting a new one so overlapping clicks can't race.
Suggested command: /impeccable harden

**[P2] Header orientation sentence reads as appended rather than integrated into the existing voice.**
Why it matters: The first two sentences of the header description are third-person descriptive prose; the added third sentence switches to a colon-led "Uusi lukija:" label-style address. Minor, but the kind of seam a reader fluent in well-edited docs (Stripe/MDN-tier) would register even without being able to name it.
Fix: Rewrite as a continuation of the existing third-person voice, e.g. "...varmistamiseksi. Ennestään tuntemattomalle lukijalle suositellaan aloitusta osioista 1–2; muut voivat siirtyä suoraan tarvitsemaansa komponenttiin sivupalkista."
Suggested command: /impeccable clarify

**[P3] `.swatch-hex::before` touch-target expansion overflows slightly into the swatch-name text above and the card padding below.**
Why it matters: Geometrically confirmed — the 44px-tall invisible hit target, centered on an ~17px line box, extends roughly into the adjacent label text and card padding. Functionally harmless (clicking the overflow area still copies the correct, currently-displayed hex), but clicking what looks like inert label text or whitespace silently does something, which is a minor surprise. Does not reach into a neighboring swatch's own hit area, so no mis-click risk between different colors.
Fix: Either accept as within WCAG 2.5.5 tolerance (minimum, not maximum, target size) and add a one-line comment noting the deliberate trade-off, or constrain the pseudo-element's vertical extent to stay within `.swatch-info`'s own box.
Suggested command: /impeccable harden
