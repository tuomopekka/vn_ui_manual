## Anti-Patterns Verdict

**LLM assessment**: Palette, typography, and component CSS are restrained and avoid obvious slop (no gradients, no glassmorphism, no hero-metric blocks). But the construction pattern is scaffold-by-template: an identical numbered-heading + underline-bar repeated ~20 times, an identical dashed-border "wireframe" callout repeated ~15 times, and 12 near-identical 5-swatch ministry color grids with zero differentiation.

**Deterministic scan**: `detect.mjs --json index.html` -> exit code 2, 2 findings:
- Em-dash overuse (warning): 76 em-dashes in body copy — an AI cadence tell the LLM reviewer didn't flag.
- Numbered section markers (advisory): sequence 05, 10, 11, 12 — independently corroborates the LLM's "numbered-heading scaffold" finding. Two isolated assessments converged on the same pattern.

No clear false positives. The numbered-markers flag is softened somewhat by design.md's own section numbers carrying real structural meaning, but the identical visual treatment (underline bar under every heading) is the templated tell, not the numbering itself.

**Visual overlays**: Skipped — no browser automation tool available in this environment. Fallback signal, not a finding.

## Overall Impression

The visual design is the right register — calm, restrained, no decorative noise — and the content faithfully mirrors design.md's tokens and rules. But the page fails its own subject matter: it's a documentation site about accessibility and precise navigation that has a broken anchor link, no visible focus indicator on its own real interactive elements, and a sidebar that lies about where you are. Fix the trust mechanism (navigation + focus + live demos) before anything else.

## What's Working

1. Faithful token-to-render mapping — type specimens and color values render live with the exact CSS from design.md.
2. Copy-paste-ready a11y attribute tables — exact ARIA attributes per component in scannable two-column format.
3. Complete, correct 12-ministry color data — every ministry's full 5-step ramp present with real hex values, nothing left as placeholder.

## Priority Issues

**[P0] Broken sidebar anchor sends users to the wrong section** — Sidebar items "14. Huomionauhat ja virhesivut" and "15. Hakukenttä" both link to #s14, but the body IDs are id="s13" (Huomionauha) and id="s14" (Hakukenttä).
Why it matters: the exact "find the spec fast" task the site exists for, failing on the most basic mechanism.
Fix: change the sidebar link to href="#s13".
Suggested command: /impeccable harden

**[P0] No visible focus indicator on the page's own interactive elements** — design.md mandates a >=3px, >=3:1-contrast orange focus ring; design-system.css only applies a narrow border-color change on two form inputs.
Why it matters: a government accessibility-compliance document that fails its own headline accessibility requirement is a direct credibility problem.
Fix: add a global :focus-visible { outline: 3px solid #F18700; outline-offset: 2px; } across all interactive elements.
Suggested command: /impeccable audit

**[P1] Sidebar active-state and scroll position are static** — is-active is hardcoded to section 1; the page has zero JS, so there's no scroll-spy.
Why it matters: violates "visibility of system status" for the primary nav mechanism of a 1700-line page.
Fix: minimal IntersectionObserver to toggle .is-active on scroll.
Suggested command: /impeccable layout

**[P1] No in-page search/filter** despite 31 sidebar entries and 12 near-identical ministry blocks.
Why it matters: contradicts the "design for zero-context vendors" principle.
Fix: lightweight client-side filter above the sidebar nav.
Suggested command: /impeccable layout

**[P2] Accordion and hover/active states aren't actually interactive** — the accordion demo hardcodes one open/two display:none items with no click handler.
Why it matters: contradicts the product's own "show, don't just tell" principle for the highest-misimplementation-risk components.
Fix: add minimal JS so the accordion demo actually toggles.
Suggested command: /impeccable harden
